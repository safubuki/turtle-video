# スクリプト品質チェックリスト

scripted-skill-generator で作成する Agent Skill は、自然言語の説明よりも実行可能な処理の信頼性を優先する。以下を満たすまで、SKILL.md の説明追加だけで逃げない。

## 必須チェック

- [ ] スクリプトは生成されるスキル自身の `scripts/` フォルダに置かれている
- [ ] Windows と Linux(Ubuntu) の両方で動くランタイムを選んでいる
- [ ] ランタイム確認コマンドが SKILL.md に書かれている
- [ ] 入力、出力、終了コード、失敗時の挙動が明記されている
- [ ] 書き込み、削除、外部送信を伴う処理は `--dry-run` または確認ステップを持つ
- [ ] パス操作は `pathlib` または `node:path` を使っている
- [ ] 出力順序、CSV列順、JSON構造が安定している
- [ ] シークレットやトークンを標準出力・ログ・CSV に出さない
- [ ] 例外時に長いスタックトレースだけを出さず、ユーザーが直せるメッセージを出す
- [ ] `--help` で使い方が分かる

## 安全性チェック

禁止または明示確認が必要な処理：

- `rm -rf`、`Remove-Item -Recurse -Force` など広範な削除
- `sudo` や管理者権限を前提にした処理
- 外部 URL へのデータ送信
- `.env`、SSHキー、クラウド認証ファイルの読み取り
- 未知のパッケージを自動インストールする処理
- ユーザーに隠して行う処理

### 主張と実装の一致

SKILL.md や説明文に安全性の約束（例:「勝手にインストールしない」「dry-run が既定」）を書いたら、その通りのコードになっているか実装を必ず読んで確認する。説明だけ安全そうで実装が伴わない、という食い違いを残さない。記事や README で挙動を説明する場合も、記憶ではなく実際のスクリプトを根拠にする。

## テスト観点

最低限、以下を実行する：

```bash
python scripts/<script>.py --help
python scripts/<script>.py --dry-run <代表引数>
```

Node.js の場合：

```bash
node scripts/<script>.mjs --help
node scripts/<script>.mjs --dry-run <代表引数>
```

確認ケース：

- 代表入力で成功する
- 空入力で分かりやすく失敗する
- 存在しないファイル/ディレクトリで分かりやすく失敗する
- 出力ファイルが既にある場合の挙動が安全
- ネットワークや認証が必要な場合、未設定時に安全に止まる

## SKILL.md への記載チェック

- [ ] 「AI が判断すること」と「スクリプトが処理すること」が分かれている
- [ ] ドライランと本実行のコマンドが両方ある
- [ ] 環境がない場合はインストール確認をするよう書かれている
- [ ] スクリプト出力を AI がどう解釈するかが書かれている
- [ ] スクリプトを更新すべき条件が書かれている

## 改善方針

同じ説明を SKILL.md に何度も足したくなったら、スクリプトの引数、ヘルプ、エラーメッセージ、出力形式を改善する。トークン節約の本質は、長い手順説明を増やすことではなく、再現可能な処理を実行物に閉じ込めることです。
