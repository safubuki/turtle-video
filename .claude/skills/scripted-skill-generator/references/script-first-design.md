# スクリプトファースト設計ガイド

このリファレンスは、Agent Skill 作成時に「自然言語指示で AI に任せる部分」と「scripts/ に置いた決定的処理へ委譲する部分」を分けるための判断基準です。

## 基本方針

スクリプトは、トークン節約だけでなく再現性、速度、検証容易性を高めるために使う。AI は、仕様の整理、ユーザーとの合意、異常値の解釈、生成物のレビューなど、人間的判断を要する部分に使う。

## スクリプト化を優先する条件

| 条件 | 理由 | 例 |
|------|------|----|
| 入力と出力の形が固定されている | 実装を再利用しやすい | JSON を集計して CSV を出す |
| データ量が多い | AI に全文を読ませるとトークンが増える | 数千件の GitHub activity を集計する |
| 同じ処理を何度も行う | 毎回コードを生成するより安定する | 月次レポート用の統計処理 |
| 正確なパースが必要 | 正規化・型変換を機械的に行える | 日付、数値、CSV、ログ |
| プロジェクト固有の規約がある | チェック条件を固定できる | 禁止 import、命名規則、ファイル配置 |
| 失敗時の原因を分類できる | エラーコードや警告で戻せる | API 認証失敗、入力ファイルなし |

## 自然言語に残す条件

| 条件 | 理由 | 例 |
|------|------|----|
| 判断が主である | スクリプト化しても分岐が増えすぎる | 設計レビュー、優先順位付け |
| 入力仕様が固まっていない | 先にヒアリングが必要 | 初回の業務設計 |
| 一度きりの小作業 | スクリプト作成コストが高い | 単発の短い変換 |
| ユーザーの好みで出力が変わる | 対話しながら決める方がよい | 文体調整、説明の粒度 |
| 外部システムの状態が不安定 | スクリプトだけでは判断不能 | UI操作や人間承認が絡む作業 |

## 代表ユースケース

### GitHub アクティビティデータ取得

スクリプト化する処理：

- GitHub CLI または API から対象期間の activity を取得
- 必要フィールドだけ抽出
- 日付、リポジトリ、種別で集計
- CSV/JSON に安定した列順で出力

AI に残す処理：

- どの指標を成果として扱うかの相談
- 外れ値や欠損の解釈
- レポート本文の作成

### 統計処理後の CSV 出力

スクリプト化する処理：

- 入力ファイルの読み込み
- 欠損値や型の正規化
- 集計値、ランキング、分布の算出
- CSV の列順、文字コード、改行コードを固定して出力

AI に残す処理：

- 集計軸の選定
- 結果の意味付け
- 次の分析観点の提案

### プロジェクト固有の静的解析

スクリプト化する処理：

- 対象拡張子の走査
- 禁止パターンの検出
- 例外パスの除外
- ファイル、行、理由を一覧出力

AI に残す処理：

- 検出結果の優先度判断
- 修正方針の提案
- 既存設計との整合性確認

## 出力契約

スクリプトを設計するときは、次の契約を先に決める：

- 入力: ファイル、ディレクトリ、期間、API 認証、オプション
- 出力: CSV、JSON、Markdown、標準出力、終了コード
- 安定性: ソート順、列順、タイムゾーン、文字コード
- 失敗時: 終了コード、エラー文、部分出力の扱い
- セキュリティ: シークレットを読まない、出さない、送らない

## 判断メモ

迷った場合は、まず小さなスクリプトで「取得」「正規化」「出力」だけを固定し、判断や文章化は SKILL.md の自然言語手順に残す。スクリプト化の目的は AI を不要にすることではなく、AI に毎回同じ定型処理を再発明させないことです。
